User Story: Boarding Intake Record Creation and Processing
Story Goal
To enable seamless creation, submission, and processing of Boarding Intake records with automated downstream record creation, field auto-population, and controlled user interactions to support efficient onboarding and maintenance workflows.

Description
As a Service/Operations Agent
I want to create and submit a Boarding Intake record (manually or via Adobe form integration) with all required details and system-driven automation
In order to ensure accurate onboarding/maintenance processing, automatic record creation, and consistent data flow across related objects in Salesforce

Dependencies


Adobe Form to Salesforce (SF) integration for intake record creation


Salesforce object configuration:


Boarding Intake


Case (Canada Boarding Request)


Boarding Detail (New FID / Product Change Activation - CAN)


Boarding Product


Boarding Reference FID


Boarding Account




Record Types configuration:


Canada Onboarding (default for Boarding Intake)


Canada Boarding Request (Case)


New FID / Product Change Activation - CAN




Field mapping and auto-population logic across objects


User access configuration (Profiles, Roles, Users)


UI configuration for dynamic field visibility and buttons



Business Impact


Ensures standardized onboarding and maintenance processes


Reduces manual effort through automation and field propagation


Improves data accuracy and consistency across systems


Enables faster case creation and processing


Enhances operational efficiency and reduces processing delays



Benefits


Automated record creation reduces manual dependency


Improved user experience with guided intake forms


Dynamic field visibility ensures relevant data capture


Multi-product handling improves flexibility


Controlled submission prevents data inconsistency


Centralized customization handling ensures better tracking



Acceptance Criteria
1. Boarding Intake Record Creation
Given an agent clicks “New” on Boarding Intake OR an Adobe form is submitted
When the record is created
Then a Boarding Intake record should be automatically created in Salesforce
And record type should default to Canada Onboarding
And data should flow from Adobe form (if applicable)

2. Record Storage and Initial Population
Given the Boarding Intake record is created
When the record is saved
Then it should be stored under the Boarding Intake tab
And fields should be auto-populated based on source (Adobe/manual)

3. Submit Button Functionality
Given the agent clicks the “Submit” button
When submission is completed
Then the following records should be automatically created:


Case (Canada Boarding Request)


Boarding Detail (New FID / Product Change Activation - CAN)


Boarding Product


Boarding Reference FID


Boarding Account



4. Field Auto-Population Across Objects
Given the Boarding Intake record is submitted
When records are created
Then:


Case Information fields should populate in Case


Boarding Detail fields should populate in Boarding Detail


Boarding Product fields should populate in Boarding Product


Boarding Reference FID & Account fields should populate accordingly



5. Record Type Defaults
Given downstream records are created
When records are initialized
Then:


Case should have record type Canada Boarding Request


Boarding Detail should have record type New FID / Product Change Activation - CAN



6. Status Synchronization
Given a Boarding Intake record has a Status
When the record is submitted
Then the same Status should be reflected on the Case record

7. Ownership Propagation
Given a Boarding Intake record has an owner
When the Case is created
Then the Case should inherit the same ownership

8. Field Visibility Rules
Given specific field conditions
When the page is loaded
Then:


Abandoned Reason should be visible only when Status = Abandoned


Permissible Purpose visible only when Case Type = Onboarding


Maintenance-related fields visible only when Case Type = Maintenance and Cause = Mirror FID


Fulfillment ID should NOT be visible when Case Type = Onboarding



9. Field Renaming
Given the page layout is configured
When fields are displayed
Then:


Account Name → GTM Account Name


CIS Account Type → Account Type


Bill To Name → BILL-TO INVOICE NAME


Account → Location Account



10. Customization Handling
Given user selects customization values
When data is entered
Then system should provide:


Multi-select field for Customization


Multi-select field for Customization Codes


Conditional multi-select field for MRUL (if MRUL selected)


Consolidated text field


And consolidated field should:


Capture all customization codes


Store in single string format


Use delimiter (e.g., semicolon)



11. Multiple Product Selection
Given agent is working on Boarding Intake
When agent clicks “Add Product”
Then a new Product field should appear
And agent should be able to search and add multiple products

12. Submit vs Save as Draft
Given user is on Boarding Intake page
When user clicks:


Submit → record should process and create downstream records


Save as Draft → record should save without triggering automation



13. Post-Submission Behavior
Given the record is submitted
When submission is completed
Then the record should become non-editable

14. Access Control
Given users are configured in Users, Profiles, Roles
When accessing Boarding Intake
Then they should have visibility and edit access

15. UI Layout Sections
Given the Boarding Intake page
When displayed
Then it should include:


Case Information


Boarding Detail


Boarding Product


Boarding Reference FID & Boarding Account


System Information
