User Story: Manual Creation of Boarding Intake Record in Salesforce
Story Goal

To enable agents to manually create Boarding Intake records with default configurations, simplified navigation, and automatic ownership assignment to ensure efficient and error-free intake processing.

Description

As a Service/Operations Agent
I want to create a Boarding Intake record directly using the “New” button with predefined defaults and ownership
In order to streamline manual intake creation, eliminate unnecessary steps, and ensure accurate ownership and data capture

Dependencies
Salesforce Boarding Intake object configuration
Record Type configuration:
Default: Canada Onboarding
Page layout configuration (all sections and fields)
Field-level validations and dynamic visibility rules
Button configuration:
New
Submit
Save as Draft
User access via Profiles, Roles, Permissions
Integration readiness (Minerva, C2O, Entitlement Master – no impact)
Business Impact
Eliminates additional user steps (record type selection screen)
Improves speed and efficiency of manual record creation
Ensures correct record type usage across all manual entries
Maintains clear ownership for accountability and processing
Benefits
Faster record creation with reduced clicks
Improved user experience through direct navigation
Accurate ownership assignment reduces routing issues
Standardization of onboarding record creation
Reduced risk of incorrect record type selection
Acceptance Criteria
1. Access to Create Record

Given the agent has required permissions
When the agent navigates to the Boarding Intake tab
Then the “New” button should be visible
And the agent should be able to initiate record creation

2. Direct Record Creation Without Record Type Selection

Given the agent clicks on the “New” button
When the system processes the request
Then the user should be redirected directly to the Boarding Intake creation page
And the record type selection screen should NOT be displayed

3. Default Record Type Assignment

Given a manual Boarding Intake record is being created
When the record creation page is opened
Then the record type should be automatically set to Canada Onboarding
And the user should not be able to change it during creation

4. Owner Assignment for Manual Records

Given a Boarding Intake record is created manually
When the record is saved
Then the record owner should default to the user who created the record
And no additional assignment logic should override this behavior

5. Page Layout Visibility

Given the record creation page is loaded
When the user views the page
Then all required sections should be visible:

Case Information
Boarding Detail
Boarding Product
Boarding Reference FID & Boarding Account
System Information
6. Field Renaming and Availability

Given fields are displayed on the page
When the user views the form
Then fields should reflect updated labels:

Account Name → GTM Account Name
CIS Account Type → Account Type
Bill To Name → BILL-TO INVOICE NAME
Account → Location Account
7. Mandatory Field Validation

Given required fields are not completed
When the user attempts to save or submit
Then the system should prevent the action
And display appropriate validation messages

8. Dynamic Field Visibility

Given field visibility depends on conditions
When conditions are met
Then:

Abandoned Reason visible only when Status = Abandoned
Permissible Purpose visible only when Case Type = Onboarding
Maintenance-related fields visible only when Case Type = Maintenance and Cause = Mirror FID
Fulfillment ID should not be visible when Case Type = Onboarding
9. Data Entry and Persistence

Given the user enters data across sections
When the record is saved
Then all values should persist correctly
And no data loss should occur

10. Product Addition Functionality

Given the user wants to add products
When the user clicks “Add Product”
Then a product selection field should appear
And the user should be able to search and select products

11. Multiple Product Handling

Given multiple products are required
When the user adds more products
Then the system should allow multiple product entries
And each entry should be captured independently

12. Customization Handling

Given the user selects customization values
When data is entered
Then:

Customization and Customization Codes should be multi-select fields
MRUL field should appear conditionally
Consolidated field should store all values in delimited format
13. Save as Draft Functionality

Given the user is filling the form
When the user clicks “Save as Draft”
Then the record should be saved without triggering downstream processes
And should remain editable

14. Submit Functionality

Given all required fields are completed
When the user clicks “Submit”
Then the system should validate data
And proceed with submission

15. Post-Submission Behavior

Given the record is submitted
When submission is successful
Then the record should become non-editable
And all fields should be read-only

16. Integration Compatibility

Given existing integrations are active
When manual records are created
Then integrations (Minerva, C2O, Entitlement Master) should not be impacted
And existing functionality should remain unchanged

17. Error Handling

Given an error occurs during creation
When the system detects the issue
Then appropriate error messages should be shown
And the user should be able to correct and proceed

18. Performance

Given the user creates or saves a record
When the system processes the request
Then response time should be within acceptable limits
And user experience should not be degraded
